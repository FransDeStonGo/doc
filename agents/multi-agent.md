# Multi-Agent системы

## Overview

Multi-agent система — это несколько AI-агентов, работающих совместно. Один агент не всегда может решить всё: контекстное окно ограничено, одна задача может выполняться параллельно разными специалистами, а некоторые подзадачи лучше изолировать. Multi-agent архитектура решает эти проблемы через декомпозицию и делегирование.

> Orchestrator-Workers — ключевой паттерн для multi-agent систем по классификации Anthropic.

---

## Зачем нужны несколько агентов

**Три главные причины:**

**1. Преодоление контекстного окна**
Задача слишком большая для одного контекста. Оркестратор разбивает на части, каждый субагент решает свою часть в своём независимом контексте.

**2. Параллелизация**
Независимые подзадачи выполняются одновременно. Агент-исследователь и агент-писатель работают параллельно вместо последовательно.

**3. Специализация**
Разные задачи требуют разного поведения, инструментов или даже моделей. Агент-ревьюер с доступом только к чтению; агент-исполнитель с доступом к записи.

---

## Роли агентов

### Orchestrator (Оркестратор)

- Получает задачу от пользователя
- Анализирует и декомпозирует на подзадачи
- Делегирует субагентам через Agent tool или API
- Синтезирует результаты в финальный ответ
- Следит за прогрессом, обрабатывает сбои

**Характеристики:** чаще всего более мощная модель (Opus, Sonnet). Управляет, а не выполняет.

### Sub-agent (Субагент)

- Получает конкретную, ограниченную задачу
- Работает в изолированном контексте
- Имеет ограниченный набор инструментов (только то, что нужно)
- Возвращает структурированный результат оркестратору

**Характеристики:** может быть более дешёвой моделью (Haiku, Sonnet). Выполняет, а не управляет.

### Типичные специализации субагентов

| Субагент | Задача | Инструменты |
|---------|--------|------------|
| **Researcher / Explore** | Поиск и анализ информации | Read, Grep, Glob, WebSearch |
| **Planner** | Проектирование архитектуры решения | Read (только) |
| **Executor** | Выполнение изменений | Edit, Write, Bash |
| **Reviewer** | Проверка качества | Read, Grep |
| **Summarizer** | Компрессия и синтез | Read |

## Изоляция контекста

Каждый субагент запускается с **чистым контекстом** — без истории оркестратора. Это ключевое свойство:

```
Оркестратор (контекст: вся задача + переписка)
    ├─ Sub-agent 1 (контекст: только подзадача A)
    ├─ Sub-agent 2 (контекст: только подзадача B)
    └─ Sub-agent 3 (контекст: только подзадача C)
```

**Зачем изоляция:**
- Не тратим токены оркестратора на детали подзадачи
- Субагент не путается в нерелевантном контексте
- Параллельные субагенты не мешают друг другу
- Контроль: субагент видит только то, что ему нужно

**Передача контекста**: оркестратор явно передаёт только нужное — конкретный запрос, файлы, данные. Результат субагент возвращает тоже явно.

## Коммуникация между агентами

### Orchestrator → Sub-agent

Оркестратор запускает субагента с промптом. В Claude Code — через Agent tool:

```python
# Оркестратор делегирует задачу субагенту
Agent(
    description="Explore codebase for auth patterns",
    subagent_type="Explore",
    model="haiku",
    prompt="""
    Найди все места где используется JWT-авторизация.
    Рабочая директория: /root/project
    Верни: список файлов и краткое описание паттерна в каждом.
    """
)
```

### Sub-agent → Orchestrator

Субагент возвращает результат как текст в своём финальном ответе. Оркестратор получает его как результат вызова инструмента. Форматы:
- Структурированный текст / markdown
- JSON для машиночитаемых данных
- Простой текст для нарративных результатов

## Паттерны multi-agent систем

### 1. Sequential (Последовательный)

```
Orchestrator → Agent A → результат → Agent B → результат → Synthesize
```

Agent B использует результат Agent A. Когда следующий шаг зависит от предыдущего.

### 2. Parallel (Параллельный)

```
Orchestrator → Agent A ─┐
             → Agent B ─┼→ Synthesize
             → Agent C ─┘
```

Независимые задачи выполняются одновременно. Оркестратор ждёт всех и синтезирует.

### 3. Hierarchical (Иерархический)

```
Orchestrator
    └─ Sub-orchestrator A
           ├─ Worker A1
           └─ Worker A2
    └─ Sub-orchestrator B
           ├─ Worker B1
           └─ Worker B2
```

Для очень сложных задач с несколькими уровнями декомпозиции. Используется редко — сложно отлаживать.

### 4. Specialist Pool (Пул специалистов)

```
Orchestrator → [Router] → Specialist 1 (код)
                        → Specialist 2 (данные)
                        → Specialist 3 (исследование)
```

Задача классифицируется и направляется к подходящему специалисту. Вариант Routing + Multi-agent.

## Multi-agent в Claude Code

Claude Code реализует sub-agents через Agent tool. Встроенные типы:

| Тип | Специализация | Рекомендуемая модель |
|-----|--------------|---------------------|
| `Explore` | Быстрое исследование кодовой базы | Haiku (экономия) |
| `Plan` | Проектирование решений | Sonnet/Opus |
| `general-purpose` | Универсальные задачи | По умолчанию |

```
# Запуск Explore-субагента на Haiku
Agent(subagent_type="Explore", model="haiku", prompt="...")
```

Каждый субагент работает в изолированном worktree или чистом контексте, не мешая основной сессии.

## Tradeoffs

**Плюсы:**
- Параллелизация ускоряет выполнение больших задач
- Специализация повышает качество на конкретных подзадачах
- Изоляция контекста снижает "шум" и экономит токены
- Разные модели для разных задач оптимизируют стоимость

**Минусы / риски:**
- **Coordination overhead** — синхронизация результатов стоит токенов и времени
- **Compounding errors** — ошибка субагента A влияет на работу субагента B
- **Потеря контекста** — при изоляции оркестратор может передать недостаточно данных
- **Сложность отладки** — тяжелее отследить где возникла проблема

**Золотое правило:** multi-agent оправдан когда задача явно требует параллелизации или специализации. Для простых задач — один агент надёжнее и дешевле.
