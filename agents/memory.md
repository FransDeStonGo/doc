# Memory

## Overview

Память агента — это то, что он знает и помнит при выполнении задачи. Anthropic выделяет четыре типа хранения, которые отличаются по персистентности, скорости доступа и применению. Правильный выбор типа памяти напрямую влияет на стоимость, скорость и качество агента.

---

## Четыре типа памяти

```
┌─────────────────────────────────────────────────────────────┐
│                    Типы памяти агента                        │
├────────────────┬──────────────┬──────────────┬──────────────┤
│  In-Context    │   External   │  In-Weights  │   In-Cache   │
│  (контекст)    │  (внешнее)   │  (в модели)  │  (кэш)       │
├────────────────┼──────────────┼──────────────┼──────────────┤
│ Текущая беседа │ БД, файлы,   │ Fine-tuning  │ Prompt cache │
│ Системный      │ vector store │ Pre-training │ KV cache     │
│ промпт         │              │              │              │
└────────────────┴──────────────┴──────────────┴──────────────┘
```

### 1. In-Context Storage

**Что это:** всё, что находится в активном контекстном окне — история беседы, системный промпт, загруженные документы, результаты инструментов.

**Характеристики:**
- Мгновенный доступ — модель видит всё сразу
- Временная — исчезает после завершения сессии
- Ограниченная — 200k токенов у Claude (≈ 150k слов)
- Дорогая — каждый токен стоит денег на каждом вызове

**Когда использовать:**
- Короткие задачи в пределах одной сессии
- Контекст, нужный модели прямо сейчас
- Результаты инструментов, history разговора

**Практика:** при заполнении контекста — применяй summarization (сжимай старые части) или sliding window (удаляй старейшие сообщения).

---

### 2. External Storage

**Что это:** данные вне модели — базы данных, файловые системы, vector stores. Агент получает доступ через инструменты: поиск, чтение, запись.

**Характеристики:**
- Персистентная — данные живут между сессиями
- Неограниченная — хранишь сколько нужно
- Требует retrieval — модель не "видит" всё сразу, а запрашивает

**Два режима доступа:**

| Режим | Как работает | Когда применять |
|-------|------------|----------------|
| **Semantic search** | Векторный поиск по смыслу | Неструктурированные данные, документы, знания |
| **Exact lookup** | Поиск по ключу, SQL | Структурированные данные, ID, факты |

**Примеры:**
- Vector store (Pinecone, Chroma, pgvector) — база знаний, документация
- SQL / NoSQL БД — данные пользователей, история транзакций
- Файловая система — конфиги, артефакты, промежуточные результаты
- Redis / KV-хранилище — быстрый кэш, сессии

**Когда использовать:**
- Данных больше чем влезает в контекст
- Нужна персистентность между сессиями
- Разные агенты должны разделять данные

---

### 3. In-Weights Storage

**Что это:** знания и поведение, запечённые в веса модели через pre-training или fine-tuning.

**Характеристики:**
- Постоянная — не меняется во время инференса
- Неявная — модель "знает", но не "помнит" источник
- Требует переобучения для обновления

**Когда использовать:**
- Специфическое доменное поведение (стиль, формат, терминология)
- Знания, которые не меняются
- Задача требует глубокой специализации

**Практика:** большинство production-агентов не использует fine-tuning — дорого, долго, сложно обновлять. Лучше решать через системный промпт + RAG.

---

### 4. In-Cache Storage (Prompt Caching)

**Что это:** кэш KV-состояния для повторяющихся частей промпта. Anthropic поддерживает prompt caching — если начало промпта идентично между запросами, платишь только за новую часть.

**Характеристики:**
- TTL 5 минут (Anthropic)
- Работает только для идентичного префикса промпта
- Снижает стоимость до 90% и latency до 85% для кэшированной части

**Когда использовать:**
- Длинный системный промпт, не меняющийся между запросами
- Большой документ, к которому обращаются многократно
- Много-шаговый агент с одинаковым контекстом на каждом шаге

**Пример:**
```python
# Помечаем длинный системный промпт для кэширования
messages=[{
    "role": "user",
    "content": [{
        "type": "text",
        "text": long_document,
        "cache_control": {"type": "ephemeral"}
    }]
}]
```

| | In-Context | External | In-Weights | In-Cache |
|--|:----------:|:--------:|:----------:|:--------:|
| **Персистентность** | Сессия | Постоянная | Постоянная | 5 мин |
| **Скорость доступа** | Мгновенная | Средняя | Мгновенная | Быстрая |

| **Обновляемость** | Легко | Легко | Сложно | Авто |
| **Стоимость** | Высокая | Низкая | Разовая | Низкая |

## Memory в Claude Code

Claude Code реализует файловую систему памяти между сессиями. Файлы хранятся в `~/.claude/projects/<path>/memory/`.

| Тип | Содержание |
|-----|-----------|
| `user` | Профиль пользователя, предпочтения, контекст роли |
| `feedback` | Правила и коррекции поведения агента |
| `project` | Текущие задачи, решения, цели проекта |
| `reference` | Ссылки на внешние ресурсы, где что искать |

Это пример External Storage с семантической типизацией — что хранить, агент решает сам.

## Паттерн: многоуровневая память

Типичная production-архитектура комбинирует типы:

```
Входящий запрос
      ↓
[Retrieve] — semantic search по vector store → добавить в контекст
      ↓
[In-Context] — системный промпт + история + retrieved данные
      ↓
[LLM] — генерирует ответ + tool calls
      ↓
[Store] — важные результаты → external storage для следующих сессий
      ↓
[Cache] — длинный системный промпт → prompt cache
```

## Tradeoffs

**Самая частая ошибка:** пытаться всё держать in-context. Это дорого и ограничено по объёму.

**Правило выбора:**
- Нужно прямо сейчас → in-context
- Нужно повторно или много данных → external storage
- Нужна скорость и повторяющийся контекст → in-cache
- Нужна специализация поведения → in-weights (редко)
