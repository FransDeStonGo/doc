# Что такое агент

## Overview

По классификации Anthropic, **agentic systems** — общий термин для всех систем, где LLM решает задачи с использованием инструментов и многошаговых рассуждений. Внутри этого термина два принципиально разных подхода: **workflow** и **agent**. Разница — в том, кто управляет процессом.

> Источник: Anthropic, "Building Effective Agents" (2024)

---

## Базовый строительный блок — Augmented LLM

Любая агентная система строится из одного базового элемента:

```
┌─────────────────────────────────┐
│         Augmented LLM           │
│                                 │
│  Memory ←→ [ LLM ] ←→ Tools    │
│              ↕                  │
│           Retrieval             │
└─────────────────────────────────┘
```

| Компонент | Роль |
|-----------|------|
| **Tools** | Внешние действия: вызов API, выполнение кода, запись файлов |
| **Memory** | Контекст: что хранить, откуда брать при следующем вызове |
| **Retrieval** | Поиск релевантной информации по запросу модели |

Anthropic рекомендует использовать **MCP (Model Context Protocol)** как стандартный способ подключения инструментов к модели.

## Agents vs Workflows

| | Workflow | Agent |
|--|----------|-------|
| **Кто управляет** | Разработчик (код) | Модель (LLM) |
| **Пути выполнения** | Предопределены заранее | Определяются динамически |
| **Число шагов** | Известно до запуска | Неизвестно до завершения |
| **Предсказуемость** | Высокая | Низкая |
| **Стоимость ошибки** | Изолирована | Накапливается (compounding errors) |
| **Подходит для** | Чётко структурированных задач | Открытых, непредсказуемых задач |

**Workflow** — это "predefined code paths": LLM вызывается в строго определённых точках, но общая логика прописана в коде. Модель не решает, что делать дальше.

**Agent** — LLM сам динамически решает: какой инструмент вызвать, нужен ли следующий шаг, когда остановиться. Разработчик задаёт только цель и доступные инструменты.

## Когда выбирать что

### Начинай с простого

Anthropic явно рекомендует: **не усложнять без необходимости**.

```
один LLM + хороший промпт
    ↓ если недостаточно
один LLM + retrieval / examples в контексте
    ↓ если недостаточно
workflow (предсказуемая задача)
    ↓ если недостаточно
agent (открытая задача)
```

### Workflow — когда

- Задача имеет предсказуемую структуру и известные шаги
- Важна высокая надёжность и контроль
- Ошибка на одном шаге не должна затрагивать другие
- Нужно минимальное потребление токенов

### Agent — когда

- Задача открытая: число шагов невозможно предсказать заранее
- Нужна адаптация к результатам промежуточных шагов
- Есть возможность запустить в изолированной среде
- Готовность принять более высокую стоимость и риск ошибок

---

## Принципы дизайна (Anthropic)

**1. Simplicity** — простота прежде всего.
Каждый добавленный компонент — это новый источник ошибок. Усложнять только когда более простой подход доказано недостаточен.

**2. Transparency** — шаги планирования должны быть видны.
Агент должен явно показывать, что он собирается делать. Это упрощает отладку и повышает доверие.

**3. Tool quality** — качество инструментов важнее промпта.
При разработке SWE-bench агента Anthropic потратили больше времени на оптимизацию инструментов, чем на основной промпт. Инструменты должны быть интуитивны, хорошо задокументированы, с защитой от ошибок (poka-yoke).

---

## Tradeoffs агентов

**Риски:**
- **Compounding errors** — ошибки накапливаются: неверный шаг влечёт неверные следующие
- **Высокая стоимость** — больше шагов = больше токенов
- **Непредсказуемость** — сложнее тестировать и гарантировать результат
- **Blast radius** — автономный агент может сделать что-то нежелательное

**Когда риски оправданы:**
- Задача действительно не поддаётся фиксированному пайплайну
- Есть возможность изолировать агента (sandbox, ограниченные permissions)
- Метрики успеха чётко определены и проверяемы (например, автотесты)
