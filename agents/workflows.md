# Workflow-паттерны

## Overview

Workflow — это агентная система с предопределёнными путями выполнения. Разработчик управляет логикой через код, LLM выполняет конкретные шаги. Anthropic выделяет 5 паттернов, которые покрывают большинство практических сценариев.

Workflows — не упрощённая версия агентов. Для большинства production-задач это правильный выбор: предсказуемы, надёжны, дешевле в токенах.

> Источник: Anthropic, "Building Effective Agents" (2024)

---

## 1. Prompt Chaining

**Суть:** разбить задачу на последовательные шаги. Выход одного вызова — вход следующего. Между шагами — опциональные gate-проверки.

```
Вход → [LLM₁] → промежуточный результат → [Gate?] → [LLM₂] → ... → Выход
```

**Когда применять:**
- Задача распадается на чёткие подзадачи
- Каждый шаг требует отдельного промпта или валидации
- Важна точность на каждом этапе (gate может остановить pipeline при ошибке)

**Примеры:**
- Написание маркетингового текста → перевод на другой язык
- Генерация плана документа → проверка плана → написание по плану
- Извлечение данных → их нормализация → финальное форматирование

**Gate-checks:** между шагами можно вставить программную проверку (не LLM). Например: если в результате шага 1 нет нужного поля — остановить и вернуть ошибку.

## 2. Routing

**Суть:** классификация входа → направление в специализированный обработчик.

```
          ┌→ Handler A (простые запросы / Haiku)
Вход → [Classifier] ─┼→ Handler B (сложные запросы / Sonnet)
          └→ Handler C (специальный случай)
```

**Когда применять:**
- Разные типы входа требуют разной обработки
- Единый промпт для всех типов даёт плохой результат
- Хочется оптимизировать стоимость (дешёвая модель для простых случаев)

**Примеры:**
- Техподдержка: общие вопросы → FAQ-агент, сложные → человек
- Маршрутизация по модели: Haiku для классификации, Sonnet для сложного reasoning
- Тип контента: текст → один пайплайн, изображение → другой

**Практика:** классификатор может быть и LLM-вызовом, и простой эвристикой — выбирать минимально сложный вариант.

## 3. Parallelization

**Суть:** несколько LLM-вызовов работают одновременно. Два варианта:

### Sectioning — параллельные независимые подзадачи

```
         ┌→ [LLM: задача A] ─┐
Вход ────┼→ [LLM: задача B] ─┼→ [Aggregator] → Выход
         └→ [LLM: задача C] ─┘
```

**Когда:** задача разбивается на независимые части. Результаты потом агрегируются.

**Примеры:**
- Проверка контента разными охранными моделями одновременно
- Анализ разных секций большого документа параллельно
- Генерация нескольких независимых компонентов системы

### Voting — одна задача, несколько попыток

```
         ┌→ [LLM: попытка 1] ─┐
Вход ────┼→ [LLM: попытка 2] ─┼→ [Voting] → Выход
         └→ [LLM: попытка 3] ─┘
```

**Когда:** нужна высокая уверенность, задача имеет правильный ответ, отдельные попытки могут ошибаться.

**Примеры:**
- Несколько проверок кода на уязвимости → принять только консенсус
- Классификация контента: 3 модели голосуют → мажоритарное решение
- Генерация ответа: несколько вариантов → выбор лучшего оценщиком

## 4. Orchestrator-Workers

**Суть:** центральная модель (оркестратор) динамически разбивает задачу на подзадачи и делегирует их воркерам. Подзадачи не предопределены — их определяет оркестратор на основе входа.

```
                   ┌→ [Worker: задача A]
Вход → [Orchestrator] ─┼→ [Worker: задача B] → [Synthesizer] → Выход
                   └→ [Worker: задача C]
```

**Чем отличается от Parallelization:** воркеры и их задачи не фиксированы в коде — оркестратор решает, что делегировать, исходя из конкретной задачи.

**Когда применять:**
- Сложные задачи с непредсказуемым набором подзадач
- Нужна специализация: разные воркеры хорошо делают разные вещи
- Задачи требуют синтеза разнородных результатов

**Примеры:**
- Coding-агент: оркестратор определяет какие файлы изменить, воркеры пишут изменения
- Research-агент: оркестратор решает какие источники искать, воркеры ищут
- Multi-step анализ: оркестратор разбивает на аспекты, воркеры анализируют каждый

**Практика:** оркестратор сам может быть более мощной моделью, воркеры — более дешёвой.

## 5. Evaluator-Optimizer

**Суть:** цикл итеративного улучшения. Один LLM генерирует, другой (или тот же) оценивает и даёт feedback. Цикл продолжается до достижения критерия.

```
Вход → [Generator] → результат → [Evaluator] → feedback
              ↑                        │
              └────── улучшить ←───────┘
                   (если не готово)
                        │
                        ↓ (если готово)
                       Выход
```

**Когда применять:**
- Есть чёткие критерии качества
- Итерация доказанно улучшает результат
- Оценщик может давать конкретный actionable feedback

**Примеры:**
- Литературный перевод: генерация → оценка точности и стиля → полировка
- Написание кода: генерация → запуск тестов → фикс по ошибкам
- Поиск информации: поиск → оценка полноты → ещё один раунд поиска

**Важно:** не злоупотреблять — каждая итерация стоит токенов. Ставить разумный лимит числа циклов.

## Decision Guide — какой паттерн выбрать

```
Задача разбивается на последовательные шаги?
  └→ ДА → Prompt Chaining

Разные типы входа требуют разной обработки?
  └→ ДА → Routing

Части задачи можно делать параллельно?
  └→ ДА → Parallelization
           ├→ Независимые подзадачи → Sectioning
           └→ Нужна уверенность → Voting

Набор подзадач зависит от конкретного входа?
  └→ ДА → Orchestrator-Workers

Нужна итеративная полировка до критерия качества?
  └→ ДА → Evaluator-Optimizer

Ничего из выше не подходит, задача открытая?
  └→ Рассмотреть полноценный Agent (см. what-is-agent.md)
```

---

## Когда Workflow лучше Agent

- Задача структурирована и предсказуема → не нужна автономия модели
- Важна надёжность и воспроизводимость результата
- Стоимость ошибки высокая — нельзя допустить compounding errors
- Нужен аудит каждого шага
- Токены дороги — workflows экономичнее

**Правило Anthropic:** начинать всегда с workflow. Переходить к agent только когда workflow явно недостаточен.
